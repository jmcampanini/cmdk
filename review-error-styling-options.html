Code review follow-up

Error rows do not always use the new error icon/color

The delegate now has explicit styling for Type == "error", but source failures are still created with their source type, such as "dir" or "action".

Current code causing the mismatch

internal/generator/root.go

func ErrorItem(src Source, err error) item.Item {
    errItem := item.NewItem()
    errItem.Type = src.Type
    errItem.Source = src.Name
    errItem.Display = fmt.Sprintf("%s error: %s", src.Name, err)
    return errItem
}

This keeps the item in the source bucket, but it means a zoxide error has Type: "dir", not Type: "error".

internal/tui/delegate.go

func (d itemDelegate) iconInfoForItem(it item.Item) iconInfo {
    if it.Type == "loading" {
        return iconInfo{icon: iconLoading, color: d.colorForSourceType(it.Data["source_type"])}
    }
    if info, ok := d.icons[it.Type]; ok {
        return info
    }
    log.Warn("no icon for item type, using fallback", "type", it.Type)
    return d.unknown
}

Because source errors are typed as dir/window/action, this returns the normal source icon/color instead of iconError and theme.Error.

Recommended fix: make errors explicit, keep source grouping in metadata

Set error rows to Type: "error", store the original source type in Data["source_type"], and update grouping to bucket error rows by that source type.

1. Change error item construction

internal/generator/root.go

func ErrorItem(src Source, err error) item.Item {
    errItem := item.NewItem()
    errItem.Type = "error"
    errItem.Source = src.Name
    errItem.Display = fmt.Sprintf("%s error: %s", src.Name, err)
    errItem.Data["source_type"] = src.Type
    return errItem
}

2. Keep error rows in the source-type bucket

internal/item/group.go

bucketKey := it.Type
if bucketKey == "loading" || bucketKey == "error" {
    if st := it.Data["source_type"]; st != "" {
        bucketKey = st
    }
}

3. Delegate code can mostly stay as-is

internal/tui/delegate.go

if info, ok := d.icons[it.Type]; ok {
    return info
}

Now source failures are actually Type: "error", so they naturally use:

"error": {iconError, t.Error},
Test updates: update root/group/model tests that currently expect source errors to have Type == "dir". Assert Type == "error" and Data["source_type"] == "dir" instead, while preserving ordering expectations.

A. Explicit error type + source metadata
Recommended		 ErrorItem sets Type: "error" and Data["source_type"]; grouping buckets error like loading. Clean model; render and non-selectable behavior align with existing Type == "error" checks. Requires test updates and care in grouping for custom source types.

Minimal patch shape for Option A

// internal/generator/root.go
func ErrorItem(src Source, err error) item.Item {
    errItem := item.NewItem()
    errItem.Type = "error"
    errItem.Source = src.Name
    errItem.Display = fmt.Sprintf("%s error: %s", src.Name, err)
    errItem.Data["source_type"] = src.Type
    return errItem
}

// internal/item/group.go
bucketKey := it.Type
if bucketKey == "loading" || bucketKey == "error" {
    if st := it.Data["source_type"]; st != "" {
        bucketKey = st
    }
}

After this, source errors render with the red error icon/color, remain non-selectable through the existing sel.Type != "error" guard, and can still appear near the source they came from.
